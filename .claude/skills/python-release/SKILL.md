# /python-release — Cut a SemVer release

## Role
Release manager — turn the accumulated changes into a clean, versioned, GitHub-ready release.

## Goal
Stamp the pending `[Unreleased]` changelog entries into a dated SemVer version block, bump the version at its single source, and keep every version location consistent.

## Deliverable
Updated `docs/release/CHANGELOG.md` (a new released block + reset `[Unreleased]`), the bumped version source, and a copy-pasteable release note.

---

## Prerequisite

The project must be loaded (`/python-load-project`, OR `docs/specs/04-architect.md` / README present). If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

Load context: `@rules/versioning.md` (+ `@rules/verification.md`).

## Step 1 — Read the current state

1. Read `docs/release/CHANGELOG.md`.
   - **Absent** → the app predates the changelog system. Stop and route to `/python-load-project` (it proposes the retroactive initialization). Do not create it here.
   - Present → read the `## [Unreleased]` section and the current top released version.
2. Read the canonical version: `config.py` `APP_VERSION`.

## Step 2 — Nothing to release?

If `[Unreleased]` has no entry: tell the user there is nothing to release (current version = X.Y.Z) and stop. Do not create an empty version block.

## Step 3 — Infer and propose the increment

Apply the inference of `@rules/versioning.md`:
1. Any `**BREAKING:**` marker or a `Removed` entry → **MAJOR**.
2. Else any `Added` entry → **MINOR**.
3. Else (only `Fixed` / internal `Changed`) → **PATCH**.

Compute the three candidate versions from the current one. Ask with `AskUserQuestion` (single question), the inferred increment **first and marked `(recommended)`**:

- `MAJOR → <x+1>.0.0` — [what counts as breaking, per `@rules/versioning.md`]
- `MINOR → <x>.<y+1>.0` — new backward-compatible feature(s)
- `PATCH → <x>.<y>.<z+1>` — fixes / internal changes only

Show the accumulated `[Unreleased]` entries above the question so the user decides with full context. The user has the final call (notably to force a MAJOR the inference cannot detect).

## Step 4 — Apply the release

On confirmation, using the chosen version `NEW` and the session date `<YYYY-MM-DD>`:

1. In `docs/release/CHANGELOG.md`: rename `## [Unreleased]` content into a new block `## [NEW] - <YYYY-MM-DD>` (keep the category order Added/Changed/Fixed/Removed/Security), and insert a fresh empty `## [Unreleased]` above it.
2. Bump the **canonical** version: `config.py` `APP_VERSION` → `NEW`. There is no mirror to sync.
3. Do **not** touch `docs/specs/`, the README, or the app `CLAUDE.md` origin (that records the framework version, not the app version).

## Step 5 — Verify + hand off

1. Apply `@rules/verification.md` version consistency: the top released version == `config.py` `APP_VERSION`; `[Unreleased]` reset empty.
2. Output the cut version block verbatim in a fenced markdown snippet — ready to paste into a GitHub release.
3. Remind the user that committing and tagging are theirs to do (this skill never commits): e.g. `git commit -am "release: vNEW"` then `git tag vNEW`.

## Anti-patterns — what NOT to do
- **Do not** create the changelog here if it is absent — route to `/python-load-project`.
- **Do not** cut a release with an empty `[Unreleased]`.
- **Do not** apply a MAJOR without explicit user confirmation.
- **Do not** bump a version location other than `config.py` `APP_VERSION` (there is no mirror).
- **Do not** rewrite or reword past released blocks — only `[Unreleased]` moves.
- **Do not** commit or tag — surface the commands, let the user run them.

## When the user asks something adjacent
- **"Just bump to X.Y.Z"** → honor an explicit version, but still move `[Unreleased]` into the dated block and update `config.py` `APP_VERSION`.
- **"Release and push to GitHub"** → do the changelog + version bump; the actual commit/tag/push stays the user's action (surface the commands).
