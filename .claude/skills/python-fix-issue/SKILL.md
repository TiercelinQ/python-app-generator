## Decision tree â€” classify the failure first

Read the actual error before touching code. Classify, then act in this order:

### 1. ruff / mypy error
- Read the exact message + `file:line`. Most are precise (unused import, type mismatch, undefined name, untyped def).
- Fix the cause. **Do not** add `# noqa` / `# type: ignore` / `Any` to silence it unless genuinely wrong and justified in a comment.

### 2. Syntax / import error (won't start)
- `ImportError`/`ModuleNotFoundError` â†’ wrong import path, missing `__init__.py`, or a missing dependency (check `requirements.txt`).
- Circular import â†’ likely an MVC violation (Model importing a View). Fix the direction, not with a local import band-aid.

### 3. Runtime exception
- Caught by `sys.excepthook` (logged critical + danger toast)? Read `logs/[app].log` for the stacktrace.
- Identify the layer: SQLite/file I/O/validation â†’ model; wiring/`try/except` â†’ controller; widget/signal/state â†’ view.
- Business path: ensure the model raises a **named** exception and the controller maps it to a `show_toast` of the right type.
- Fix at the layer that owns the cause, not where the symptom surfaces.

### 3b. Salesforce CLI failure (if the integration is on)
- `SfCliNotFoundError` â†’ `sf` not found or not launchable. On Windows, a `sf.cmd` shim (`npm i -g`) is not launchable by `subprocess` shell=False â€” point to the standalone `sf.exe` installer or set `SF_CLI_PATH`.
- `SfCommandError` "rĂ©ponse sf illisible" â†’ CLI version drift or a command run without `--json`; verify the flags against the matching `sf-cli-reference/` section (`sf <cmd> --help`).
- `SfCommandError` with a `status != 0` message â†’ read it, surface via `show_toast("danger", ...)`; ensure the call goes through `models/sf_cli.py` with an args list (never `shell=True`). See @rules/sf-cli.md.

### 4. DB / migration issue
- `DatabaseError` "base en version X, code attend Y" â†’ schema ahead of code; check `config.DB_SCHEMA_VERSION` vs `MIGRATIONS`.
- New column/table missing â†’ add a `MIGRATIONS[n]` entry and bump `DB_SCHEMA_VERSION` (up-only). Never an automatic `down`.
- Connection error â†’ must go through `models/db.py` `get_connection()`; no `connect` elsewhere.

### 5. QSS / design deviation (visual bug)
- Hardcoded color/size in Python, missing `objectName`, partial dark-mode override, `border-radius`/shadow/gradient, `QMessageBox` for a business message.
- Route the value through `styles_*.qss` (or `config.ICON_COLORS` for icons), add the missing `objectName`, replace `QMessageBox` with a toast / styled `QDialog`. See `@rules/qss.md` / `@rules/errors.md`.

## Steps

Read `design-system.md` / `layout.md` on demand if the fix touches UI (no longer auto-imported).

1. Get the real error (command output, `logs/`, traceback, or the user's repro). If you cannot reproduce, ask for the exact message â€” do not guess.
2. Classify with the tree above; read both the failing site and the declaration it depends on.
3. **Ask: "knowing what I know now, what is the clean fix?"** Implement that, not the quickest patch. If the clean fix is larger than expected, say so before applying.
4. Apply the minimum correct change. Respect the layers, the contract (`docs/specs/04-architect.md`), and the rules.
5. **Verify**: re-run `@rules/verification.md Â§A` for the affected area; confirm the targeted failure is gone and nothing else broke. Then apply `@rules/readme.md` â€” if the fix changed a README-documented aspect (DB schema/migration, dependency, structure), regenerate the README.
6. If it took several attempts: produce the **cleanup report** (`@rules/mvc.md` â€” list every dead element added during failed attempts), then offer `Do you want to remember this point? /python-save-memory`.

## Anti-patterns â€” what NOT to do
- **Do not** silence a ruff/mypy finding with `# noqa` / `# type: ignore` / `Any` instead of fixing the cause.
- **Do not** fix a circular import with a function-local import when the real cause is an MVC direction violation.
- **Do not** wrap a bug in `except Exception: pass` to hide it.
- **Do not** hardcode a value to "make it look right" â€” fix it through the QSS/tokens.
- **Do not** write an automatic `down` migration or bypass `models/db.py`.
- **Do not** leave failed-attempt residue in the file. Clean it up.
- **Do not** expand the fix into an unrequested refactor; flag genuinely-wrong surrounding code for `/python-refactor-code`.
- **Do not** claim it is fixed without re-running the verification (or telling the user which command to run if no env).

## When the user asks something adjacent
- **"Just make the error go away"** â†’ the error is a symptom; explain you are fixing the cause. A genuine temporary silence is explicit and flagged as debt.
- **"It's slow"** â†’ not a crash; optimization (query shape, signal storms, heavy `__init__`) â€” handle as a finalization or `/python-refactor-code` item, on request.
